# Data Model: Auto-Create Page After 3 Questions

**Feature**: 003-auto-page-creation  
**Date**: 2025-01-17  
**Status**: Complete

## Overview

This feature does not introduce new database entities or modify existing schemas. It leverages the existing Lab, LabPage, and Question entities from the core system. The auto-page-creation logic is implemented as a workflow enhancement at the application layer.

---

## Existing Entities (No Changes)

### Lab Entity

**Purpose**: Represents a lab that can contain multiple pages. Already implemented in the system.

**Source**: `@whatsnxt/core-types` (workspace package)

**Key Fields**:
```typescript
interface Lab {
  id: string;              // UUID primary key
  name: string;            // Lab title
  description?: string;    // Lab description
  labType: string;         // Type of lab
  architectureType: string; // AWS, Azure, GCP, etc.
  instructorId: string;    // Instructor who owns the lab
  status: 'draft' | 'published'; // Publication status
  pricing?: any;           // Pricing information
  associatedCourses?: string[]; // Associated course IDs
  createdAt: Date;
  updatedAt: Date;
}
```

**Validation Rules** (enforced by backend):
- `name`: Required, min 3 characters, max 200 characters
- `instructorId`: Required, must be valid UUID
- `status`: Draft by default, can only publish if lab has valid tests
- `labType`: Required, must be one of allowed types
- `architectureType`: Required, must be one of: AWS, Azure, GCP, Common, Hybrid

**State Transitions**:
```
[New Lab] → draft → published
          ↓
       (can be deleted if draft)
       (cannot be deleted if published)
```

**Usage in Feature**: 
- Auto-page-creation occurs within draft labs only (published labs are read-only)
- The `labId` is required to create new pages via `labApi.createLabPage(labId, data)`

### LabPage Entity

**Purpose**: Represents a page within a lab. Each page can contain one question and/or one diagram test.

**Source**: `@whatsnxt/core-types` (workspace package)

**Key Fields**:
```typescript
interface LabPage {
  id: string;              // UUID primary key
  labId: string;           // Foreign key to Lab
  pageNumber: number;      // Sequential page number (1, 2, 3, ...)
  hasQuestion: boolean;    // Whether page has a question
  hasDiagramTest: boolean; // Whether page has a diagram test
  createdAt: Date;
  updatedAt: Date;
}
```

**Relationships**:
- **Lab** (many-to-one): Each page belongs to exactly one lab
- **Question** (one-to-one): Each page can have zero or one question
- **DiagramTest** (one-to-one): Each page can have zero or one diagram test

**Validation Rules** (enforced by backend):
- `labId`: Required, must reference existing draft lab
- `pageNumber`: Auto-generated by backend (max existing page number + 1)
- `hasQuestion`: Boolean, defaults to false
- `hasDiagramTest`: Boolean, defaults to false

**Auto-Generation Logic** (backend):
```typescript
// When creating a page, backend automatically calculates next page number
const maxPageNumber = await LabPage.max('pageNumber', { where: { labId } });
const newPageNumber = (maxPageNumber || 0) + 1;
```

**Usage in Feature**:
- Auto-creation calls `labApi.createLabPage(labId, {})` with empty data object
- Backend generates the next sequential page number automatically
- Frontend navigates to the new page: `/labs/${labId}/pages/${newPageId}`

### Question Entity

**Purpose**: Represents a question (MCQ, True/False, Fill in the blank) on a lab page.

**Source**: `@whatsnxt/core-types` (workspace package)

**Key Fields**:
```typescript
interface Question {
  id?: string;             // UUID primary key (undefined for new questions)
  labPageId: string;       // Foreign key to LabPage
  type: 'MCQ' | 'True/False' | 'Fill in the blank';
  questionText: string;    // Question prompt
  options?: Array<{ text: string }>; // Options for MCQ
  correctAnswer: string;   // Correct answer
  createdAt?: Date;
  updatedAt?: Date;
}
```

**Relationships**:
- **LabPage** (many-to-one): Each question belongs to exactly one page

**Validation Rules** (enforced by backend):
- `labPageId`: Required, must reference existing page
- `type`: Required, must be one of the allowed types
- `questionText`: Required, min 10 characters, max 1000 characters
- `options`: Required for MCQ and True/False, must have at least 2 options
- `correctAnswer`: Required, must match one of the options (for MCQ/True-False)

**Edit vs New Detection** (client-side):
```typescript
const isEdit = !!question.id; // If ID exists, it's an edit
const isNew = !question.id;   // If ID is undefined, it's new
```

**Usage in Feature**:
- The 3-question threshold counts only NEW questions (`!question.id`)
- Editing existing questions (with `question.id`) does not count toward threshold
- The `saveQuestion` API accepts optional `questionId` parameter for updates

## Application-Layer State Management

### Auto-Page-Creation State

**Purpose**: Track question count and auto-creation status within the custom hook.

**Implementation**: React hook state (ephemeral, not persisted)

**State Shape**:
```typescript
interface AutoPageCreationState {
  questionCount: number;      // Number of questions on current page (0-N)
  isCreatingPage: boolean;    // Loading state during page creation
  error: Error | null;        // Error state if page creation fails
}
```

**State Lifecycle**:
1. **On Page Load**: Fetch current page via `labApi.getLabPageById(labId, pageId)` to get initial question count
2. **On Question Save**: Increment count if new question (`!isEdit`), check if count reaches 3
3. **On Auto-Creation Trigger**: Set `isCreatingPage = true`, call API, navigate on success
4. **On Page Navigation**: Reset state when `pageId` changes (useEffect cleanup)

**State Reset Trigger**:
```typescript
useEffect(() => {
  // Reset count when navigating to a different page
  setQuestionCount(0);
  setError(null);
  
  // Fetch initial count for the new page
  labApi.getLabPageById(labId, currentPageId).then(page => {
    // Count questions on the page (simplified logic)
    const count = page.data.question ? 1 : 0;
    setQuestionCount(count);
  });
}, [currentPageId]); // Re-run when page changes
```

**Why Ephemeral State?**
- Question count is derived from backend data (no need to persist)
- Resets correctly when navigating between pages
- Simpler than global state (no need for Redux/Context)
- Aligns with React 19 local state patterns

## Data Integrity Considerations

### Question Save Before Page Creation

**Requirement**: The 3rd question MUST be persisted before attempting page creation.

**Implementation**: Sequential operations (not atomic transaction)

**Flow**:
```typescript
// 1. Save question (MUST succeed)
const savedQuestion = await labApi.saveQuestion(labId, pageId, questionData);
// ✅ Question is now in database

// 2. Update local count
setQuestionCount(prev => prev + 1);

// 3. Attempt page creation (may fail, question is safe)
if (newCount === 3 && !isEdit) {
  try {
    const newPage = await labApi.createLabPage(labId, {});
    // ✅ Page created, navigate
  } catch (error) {
    // ❌ Page creation failed, but question is saved (no data loss)
  }
}
```

**Why Sequential?**
- Simpler implementation (no backend changes)
- Data safety: question always saved first
- Failure isolation: page creation failure doesn't affect question
- Manual fallback available: instructor can create page manually

**Trade-offs**:
- Not atomic (page creation can fail while question succeeds)
- Acceptable: spec requires data safety, not atomicity
- UX compensates: error message guides user to manual page creation
